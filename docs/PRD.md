# PRD - 반도체 시료 생산주문관리 시스템

> 본 문서는 시스템 개요, 역할별 흐름도, 주문 상태 흐름, 기능별 요구사항 자료를 기반으로 정리한 기능 명세서입니다.

## 1. 배경 및 목적

S-Semi는 다양한 반도체 시료(Sample)를 생산하여 연구소, 팹리스 업체, 대학 연구실 등 고객에게 납품한다.
주문량 증가로 엑셀·메모장 기반 관리의 한계(주문 처리 여부 파악 불가, 공정 완료 시점 불명확, 불필요한 추가 생산 등)가
드러났고, 이를 해결하기 위해 콘솔 기반의 "반도체 시료 생산주문관리 시스템"을 개발한다.

## 2. 시스템 개요

- 콘솔(CLI) 기반으로 동작하며, 담당자가 메뉴를 통해 직접 명령을 입력한다.
- 생산 라인(Production Line)은 공장에서 시료 하나를 생산하는 설비 흐름이며, 단일 라인으로 동작한다.
- 생산 라인은 주문이 들어온 시료에 대해서만, 한 번에 하나씩 생산한다.
- 시료는 시스템에 등록된 것만 주문 가능하다.

## 3. 역할(Actor)

| 역할 | 설명 |
|---|---|
| 고객 | 필요한 시료를 이메일 등으로 요청하는 시료 요청자 (시스템 외부, 직접 접근 없음) |
| 주문 담당자 | 고객 요청에 맞게 주문서를 작성/관리 (시료 주문 등록) |
| 생산 담당자 | 개발 시료 등록, 주문 수신 후 승인/거절, 생산·출고 처리 |

흐름: 고객 → (시료 요청) → 주문 담당자 → (주문서 전달) → 생산 담당자 → (승인/거절) → 주문 담당자

## 4. 주문 상태(Order Status)

모든 주문은 아래 상태 중 하나를 가지며, `REJECTED`는 정상 흐름 밖의 상태로 모니터링 대상에서 제외한다.

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 외, 모니터링 제외) |
| PRODUCING | 주문 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 + 출고 대기 중 (재고 충분) |
| RELEASE | 출고 완료 |

### 상태 전이도

```
주문 등록 (RESERVED)
      │
      ▼
   승인 여부
   ├─ 거절 ─────────────────────▶ REJECTED
   └─ 승인
        ▼
     재고 확인
     ├─ 재고 충분 ──────────────▶ CONFIRMED ──(출고 처리)──▶ RELEASE ──▶ 고객 도달
     └─ 재고 부족 ──▶ PRODUCING ──(생산 완료)──▶ CONFIRMED ──(출고 처리)──▶ RELEASE
```

## 5. 기능 명세

### 5.1 메인 메뉴

기능별 선택 화면을 표시하고, 전체 시료에 대한 요약 정보(등록 시료 수, 총 재고, 전체 주문 건수, 생산라인 대기 건수 등)를
확인할 수 있다.

| 메뉴 | 설명 |
|---|---|
| 시료 관리 | 새로운 시료 등록, 목록 조회, 이름 검색 |
| 시료 주문 | 고객 주문 접수 |
| 주문 승인/거절 | 생산 라인 담당자의 승인·거절 처리 |
| 모니터링 | 상태별 주문 수 및 시료별 재고 현황 확인 |
| 생산 라인 조회 | 현재 생산 중인 시료 및 대기 중인 생산 큐 확인 |
| 출고 처리 | CONFIRMED 상태 주문에 대해 출고 실행 |

### 5.2 시료 관리

시료(Sample)는 시스템의 가장 기본이 되는 단위이며, 각 시료는 고유한 이름과 속성을 가진다.
시스템에 등록된 시료만 주문할 수 있다.

| 기능 | 설명 |
|---|---|
| 시료 등록 | 새로운 시료를 시스템에 추가. 속성 값: 시료 ID, 이름, 평균 생산시간, 수율 |
| 시료 조회 | 등록된 모든 시료 목록과 현재 재고 수량을 함께 표시 |
| 시료 검색 | 이름 등 속성으로 특정 시료를 검색 |

- 수율 = 정상적으로 생산된 시료 수 / 총 생산 시료 수
  - 예) 100개 생산 중 정상 물품 90개 → 수율 0.9

### 5.3 시료 주문

고객이 시료를 요청하면 주문 담당자가 주문을 생성한다.

- **시료 예약**: 고객이 원하는 시료와 수량을 주문하며, 이 시점의 주문 상태는 `RESERVED`.
- **예약 시 입력 값**: 시료 ID, 고객명, 주문 수량

### 5.4 주문 승인/거절

접수된 주문(`RESERVED`) 목록을 확인하고, 특정 주문에 대해 승인 혹은 거절을 처리한다.

- **접수된 주문 목록**: `RESERVED` 상태의 주문 목록을 표시
- **주문 승인**: 접수된 특정 주문을 승인
  - 승인 시 재고 상황에 따라 2가지 방식으로 자동 처리
    - 재고가 충분한 경우 → 주문을 즉시 `CONFIRMED` 상태로 전환
    - 재고가 부족한 경우 → 생산 라인에 자동으로 등록하고, 주문 상태를 `PRODUCING`으로 전환
- **주문 거절**: 접수된 특정 주문을 거절하며, 즉시 `REJECTED` 상태로 전환

### 5.5 생산 라인

생산 라인에 대한 정보를 표시한다. 주문량에 대한 부족분을 생산하되, 수율 및 오차를 고려하여 시료를 생산한다.

- 부족분 = 주문 수량 − 현재 재고
- 실 생산량 = ceil(부족분 / 수율)
- 총 생산 시간 = 평균 생산시간 × 실 생산량
- 생산 완료 시 주문 상태를 `PRODUCING` → `CONFIRMED` 로 변경

**생산 현황 표기**: 현재 생산 중인 시료에 대한 정보(주문 정보, 현재까지의 생산량 등)를 표기하며, 표기 정보 수준은
구현 시 자율적으로 결정한다.

**대기 주문 확인**: 생산 라인의 대기열인 생산 큐(Queue)를 이용해 생산 작업을 대기 중인 목록을 출력한다.
- 스케줄링 전략: FIFO(선입선출)

### 5.6 출고 처리

재고가 충분해진 `CONFIRMED` 주문에 대하여 출고를 처리한다.

- 특정 주문에 대해 출고를 실행
- 처리 후 주문 상태가 `RELEASE`로 전환

### 5.7 모니터링

주문에 따른 시료 재고 정보를 모니터링하며, 담당자가 현재 시스템 상태를 한눈에 파악할 수 있도록 구성한다.

- **주문량 확인**: 현재 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 주문 목록을 확인. `REJECTED`는
  유효한 주문이 아니므로 집계에서 제외.
- **재고량 확인**: 각 시료별 현재 재고 수량을 확인하며, 주문 대비 재고 수량에 따라 상태를 함께 표기.
  - 여유: 주문 대비 재고 충분 상태
  - 부족: 주문 대비 재고 수량 부족 상태
  - 고갈: 재고 수량이 0인 상태

## 6. 데이터 모델(요약)

### 시료 (Sample)

| 필드 | 설명 |
|---|---|
| 시료 ID | 시료 고유 식별자 (예: S-001) |
| 이름 | 시료명 (예: 실리콘 웨이퍼-8인치) |
| 평균 생산시간 | 개당 평균 생산 소요 시간 (min/ea) |

| 현재 재고 | 현재 보유 중인 시료 수량 (ea) |

### 주문 (Order)

| 필드 | 설명 |
|---|---|
| 주문번호 | 주문 고유 식별자 (예: ORD-20260416-0043) |
| 시료 ID | 주문 대상 시료 |
| 고객명 | 주문 고객 |
| 주문 수량 | 요청 수량 (ea) |
| 상태 | RESERVED / REJECTED / PRODUCING / CONFIRMED / RELEASE |

## 7. 데이터 영속성

데이터 저장은 `../PoC/DataPersistence`(Python 라이브러리)를 이용하여 영속성을 유지하도록 구현한다.

- 표준 라이브러리 `json` 모듈만 사용하는 외부 의존성 없는 CRUD 라이브러리이며, `pip install -e ../PoC/DataPersistence`로
  설치 후 `from data_persistence import DataPersistence`로 사용한다.
- 레코드 배열(`list[dict]`)을 파일 하나에 저장하며, 파일이 없으면 최초 접근 시 자동 생성한다. 저장은 임시 파일 +
  `os.replace`를 이용한 원자적 쓰기로 처리되어 손상을 방지한다.
- 시료(Sample), 주문(Order) 각각을 별도의 JSON 파일(예: `samples.json`, `orders.json`)로 관리하고, 각 레코드의
  `id` 필드는 시료 ID / 주문번호를 그대로 사용한다.
- 제공 API: `create(record)`, `read_all()`, `read(key, field="id")`, `update(id, changes)`, `delete(id)`.
- 예외: `InvalidJsonError`(파일이 유효한 JSON 배열이 아님), `RecordNotFoundError`(대상 id 없음),
  `DuplicateIdError`(id 중복 생성 시도).
- 상세 사용법은 `../PoC/DataPersistence/README.md`, 구현 결정 사항은 `../PoC/DataPersistence/CLAUDE.md` 참고.

## 8. 프로젝트 구조 (MVC)

프로젝트 구조는 `../PoC/ConsoleMVC`의 `mvc-skeleton` Skill을 활용해 MVC(Model-View-Controller) 구조로 구현한다.

- `../PoC/ConsoleMVC/.claude/skills/mvc-skeleton/`를 본 프로젝트의 `.claude/skills/mvc-skeleton/`로 복사하여
  Skill을 적용하고, "MVC 스켈레톤 만들어줘" 요청으로 아래 골격을 생성한다.

```
project-root/
├── app/
│   ├── __init__.py
│   ├── models/
│   │   └── __init__.py        # BaseModel — 시료/주문 모델
│   ├── views/
│   │   └── __init__.py        # BaseView  — 콘솔 메뉴/출력
│   └── controllers/
│       └── __init__.py        # BaseController — 메뉴별 처리 흐름
├── config.py
├── main.py
└── requirements.txt
```

- Model: 시료(Sample), 주문(Order) 데이터와 `DataPersistence` 기반 영속성 처리 담당.
- View: 콘솔 메뉴 표시, 사용자 입력/출력 담당 (메인 메뉴, 시료 관리, 시료 주문, 주문 승인/거절, 생산 라인, 출고 처리,
  모니터링 각 화면).
- Controller: 메뉴 선택에 따른 흐름 제어, Model과 View 연결, 주문 상태 전이·생산 큐(FIFO) 처리 등 비즈니스 로직 담당.
- Skill 생성 파일은 최소 stub 수준이며, 실제 도메인 로직은 본 PRD의 기능 명세에 맞춰 구현한다.
- Skill 상세 내용은 `../PoC/ConsoleMVC/.claude/skills/mvc-skeleton/SKILL.md`, 설계 배경은
  `../PoC/ConsoleMVC/docs/PLAN.md` 참고.

## 9. 개발 방법론 (TDD)

각 기능 구현 단계는 `.claude/skills/tdd-skill/SKILL.md`에 정의된 테스트 주도 개발(TDD) 방법을 활용한다.

- 원칙: 실패하는 테스트가 먼저 존재하지 않는 한 프로덕션 코드를 작성하지 않는다 (Red-Green-Review).
  1. **RED**: 이번에 구현할 동작 1가지를 `PLAN.md`에 작성(기존 내용은 삭제 후 새로 작성)하고, 실패하는 테스트를
     한 개 작성한 뒤 실제로 실패하는 것을 확인한다.
  2. **RED 리뷰 (사용자 검토 필수)**: `PLAN.md`와 실패 테스트를 사람 파트너(사용자)에게 반드시 검토받는다.
     사용자 승인 없이는 GREEN 단계로 넘어가지 않으며, 반려 시 피드백을 반영해 RED를 다시 수행한다. 승인 후에만
     `PLAN.md`와 실패 테스트를 커밋한다.
  3. **GREEN**: 테스트를 통과시키는 최소한의 코드만 작성하고, 전체 테스트가 통과하는지 확인한다.
  4. **REVIEW (사용자 검토 필수)**: 그린 상태를 유지하며 중복 제거·이름 개선 등 정리를 한 뒤, 구현 내용을
     사람 파트너(사용자)에게 반드시 검토받는다. 사용자 승인 없이는 구현을 커밋하지 않으며, 반려 시 피드백을
     반영해 REVIEW를 다시 수행한다. 승인 후에만 구현을 커밋한다.
- RED 리뷰와 REVIEW는 생략 불가능한 검토 게이트다 — subagent(`ai-action`, `compliance-verify`, `test-verify` 등)의
  자체 판단만으로 이 두 단계를 대체하거나 건너뛰지 않는다.
- 본 시스템의 각 기능(5장의 시료 관리, 시료 주문, 주문 승인/거절, 생산 라인, 출고 처리, 모니터링)과 데이터 영속성
  (`data_persistence` 연동), MVC 각 계층(Model/View/Controller) 구현 시 모두 위 사이클을 단위로 진행한다.
- mock은 파일시스템 등 외부 경계에서만 사용하고, 내부 로직(주문 상태 전이, 실생산량 계산, FIFO 큐 등)은 실제 코드로
  검증한다.
- 상세 절차·체크리스트·안티패턴은 `.claude/skills/tdd-skill/SKILL.md` 참고.

## 10. 구현 워크플로우 (Subagent 활용)

9장 TDD 사이클(RED → RED 리뷰 → GREEN → REVIEW)의 각 단계는 아래 4개 subagent로 진행한다. RED 리뷰와 REVIEW의
사용자 검토 필수 원칙은 9장을 그대로 따른다.

| Subagent | 역할 |
|---|---|
| `doc-consistency-verify` | 이번에 구현할 내용이 PRD/CLAUDE.md 등 문서와 일관되는지 사전 검사 (문서와 어긋나면 구현 전에 PLAN.md/문서를 먼저 정정) |
| `ai-action` | 실제 구현(TDD의 RED 테스트 작성 및 GREEN 코드 작성)을 담당 |
| `compliance-verify` | 구현 내용이 요구사항(PRD 5장 기능 명세 등)을 충분히 만족하는지 검사 |
| `test-verify` | 테스트 코드가 정상적으로 통과하는지 검사 |

### 진행 순서

1. `doc-consistency-verify`로 구현 예정 내용과 문서(PRD.md, CLAUDE.md) 간 불일치가 없는지 확인한다.
   불일치가 있으면 구현에 앞서 문서 또는 계획을 먼저 정정한다.
2. `ai-action`이 RED 단계(`PLAN.md` 작성, 실패하는 테스트 작성 및 실패 확인)를 수행한 뒤, 9장의 RED 리뷰(사용자
   검토 필수)를 거친다.
3. `ai-action`이 GREEN 단계(테스트를 통과시키는 최소 구현)를 수행한다.
4. `compliance-verify`(요구사항 충족 여부)와 `test-verify`(테스트 통과 여부)를 실행한다. 두 검사는 서로 의존성이
   없으므로 병렬 수행이 가능한 경우 병렬로 수행한다. 하나라도 실패하면 `ai-action`으로 돌아가 구현을 보완한다.
5. 두 검사를 모두 통과하면 9장의 REVIEW(사용자 검토 필수)를 거쳐 승인 후 커밋한다.

```
doc-consistency-verify → ai-action(RED) → [RED 리뷰: 사용자 검토]
    → ai-action(GREEN) → ┬→ compliance-verify ─┐
                          └→ test-verify ───────┴→ [REVIEW: 사용자 검토] → 커밋
```
